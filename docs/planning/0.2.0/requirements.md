# 电竞赛程聚合网站 0.2.0 需求文档

## 1. 文档说明

本文档用于规划 0.2.0 的产品能力升级。0.1.0 已完成个人可用的赛程聚合 MVP，核心能力是按游戏和今天/明天查看比赛。0.2.0 的重点是把页面从“今日赛程列表”升级为“可查询、可筛选、可回溯的赛事信息工具”。

## 2. 版本目标

### 2.1 核心目标

- 将产品能力从“按游戏查看比赛”升级为“按条件查询赛事与结果”。
- 支持更细粒度的赛事信息筛选，不能只按游戏项目筛选。
- 支持历史日期和自定义日期范围，方便查询过往比赛结果。
- 区分赛程和赛果，让用户能快速查看未来比赛与已结束比赛。
- 保持 0.1.0 的轻量、高效、个人工具定位。
- 保持 PandaScore token 只在后端使用，前端不直接访问外部数据源。

### 2.2 优先级

P0，本版本必须完成：

- 日期范围筛选，包括历史比赛查询。
- 比赛状态筛选：未开始、进行中、已结束、延期、取消。
- 多维筛选：游戏项目、赛事名称、队伍关键词。
- 赛程与结果两个默认视图。
- 比赛详情展开或详情弹窗。
- 旧接口兼容和基础测试覆盖。

P1，本版本建议完成：

- 赛区/地区筛选。
- 赛事阶段展示和筛选。
- 快捷日期项：本周、下周、最近 30 天、过去比赛、未来赛程。
- 更多排序方式：状态优先、按更新时间、按赛事名称。
- 直播或回放链接展示，如果数据源字段可稳定获得。

P2，后续版本优先考虑：

- 收藏或关注游戏、赛事、队伍。
- “热门赛事”和“我的关注”独立首页标签。
- 赛事重要程度或赛事级别权重。

### 2.3 非目标

- 不做用户注册、登录和多用户系统。
- 不做社区、评论、博彩、赔率或预测功能。
- 不做实时局内数据面板。
- 不做复杂后台管理系统。
- 不在本版本实现通知、日历订阅、收藏关注。
- 不在本版本实现复杂的个性化推荐或热度算法。

## 3. 用户场景

### 3.1 查询未来赛程

作为用户，我希望查看未来几天某个游戏、赛事或队伍的比赛，方便安排观赛时间。

示例：

- 查询未来 7 天的 LoL 比赛。
- 查询下周所有 CS2 国际赛事。
- 查询某支队伍近期是否有比赛。
- 查询 CS2 + 国际赛 + 未来一周的赛程。

### 3.2 查询历史赛果

作为用户，我希望查询过去某一天或某一段时间的比赛结果，方便回看比分和确认胜负。

示例：

- 查询昨天已结束的 Valorant 比赛。
- 查询 2026-05-01 到 2026-05-24 的 LoL 已结束比赛。
- 查询某个赛事最近几场比赛结果。
- 查询 Valorant + VCT + 昨天的结果。

### 3.3 聚焦重要赛事

作为用户，我希望按赛事名称、赛区或阶段筛选，避免在大量小赛事中丢失重点比赛。

示例：

- 只看 LPL、LCK、VCT、Major 等赛事。
- 只看季后赛或淘汰赛。
- 只看国际赛或指定赛区比赛。
- 查询 LoL + LPL + 已结束 + EDG 的比赛。

## 4. 功能需求

### 4.1 多维筛选

页面需要支持以下筛选维度：

- 游戏项目：全部、LoL、Dota2、CS2、VALORANT；0.2.0 可以先按当前已接入游戏落地，但接口和 UI 不应写死只能支持三个游戏。
- 比赛状态：全部、未开始、进行中、已结束、已延期、已取消。
- 赛事名称：支持从当前数据中选择或输入关键词搜索。
- 队伍名称：支持按队伍名称或缩写搜索。
- 赛区/地区：优先使用 PandaScore 可提供的国家、地区或联赛字段。
- 赛事阶段：如常规赛、小组赛、季后赛、淘汰赛、决赛；若数据源缺失，本版本可先作为可选展示字段，不强制筛选。
- 重要程度：本版本不强制实现；如果数据源有稳定字段，可以先展示为赛事级别，不做筛选。

筛选规则：

- 多个筛选条件之间采用“并且”关系。
- 同一筛选维度 MVP 可先单选；后续可扩展为多选。
- 筛选后列表仍按用户选择的排序方式展示。
- 清空筛选应能一键回到默认视图。
- 筛选条件应体现在页面状态中，刷新页面后至少保留 URL 查询参数。
- 关键词搜索需要同时命中队伍、赛事、联赛和游戏展示名。

### 4.2 日期与时间范围

日期能力需要从“今天/明天”升级为完整时间查询。

必须支持：

- 今天。
- 明天。
- 本周。
- 下周。
- 最近 7 天。
- 未来 7 天。
- 自定义单日。
- 自定义日期范围。
- 过去比赛。
- 未来赛程。

建议支持：

- 上周。
- 最近 30 天。
- 未来 30 天。

规则：

- 日期仍以北京时间 `Asia/Shanghai` 为基准。
- 后端查询外部 API 时继续转换为 UTC 范围。
- 自定义日期范围需要限制最大跨度，建议 MVP 限制为 31 天，避免 API 请求过多。
- 历史查询默认优先展示已结束比赛。
- “过去比赛”默认等价于最近 7 天 + 已结束 + 时间降序。
- “未来赛程”默认等价于今天到未来 7 天 + 未开始/进行中 + 时间升序。
- 如果用户选择快捷日期后又修改具体日期，以具体日期为准。

### 4.3 赛程与赛果视图

页面建议提供两个主要视图：

- 赛程：偏未来比赛，默认展示未开始和进行中比赛。
- 结果：偏历史比赛，默认展示已结束比赛。

页面可以预留但不强制实现两个后续标签：

- 热门赛事：后续基于赛事级别、近期热度或人工权重展示。
- 我的关注：后续基于用户关注的游戏、赛事或队伍展示。

切换规则：

- 赛程视图默认日期范围为今天到未来 7 天。
- 结果视图默认日期范围为最近 7 天。
- 用户手动选择日期范围后，视图不应覆盖用户选择。
- 用户手动选择状态后，视图不应强行恢复默认状态。

### 4.4 比赛详情

每场比赛应支持展开详情或进入详情视图。

详情字段：

- 游戏项目。
- 赛事名称。
- 联赛或系列赛名称。
- 队伍信息。
- 比赛开始时间和结束时间。
- 比赛状态。
- BO 赛制。
- 总比分。
- 胜者。
- 地图或小局结果，如果数据源提供。
- 赛事国家、赛区、奖金池、等级等扩展字段，如果数据源提供。
- 直播链接，如果数据源提供。
- 回放链接，如果数据源提供。
- 数据来源。
- 数据更新时间。

本版本不强制接入直播链接和回放链接；如果 PandaScore 字段可用，可以作为展示增强。

展示规则：

- 未开始比赛重点展示开赛时间、对阵、赛事和 BO 赛制。
- 进行中比赛重点展示状态、当前比分和最后更新时间。
- 已结束比赛重点展示最终比分、胜者和小局结果。
- 延期或取消比赛需要明确展示状态，不能只显示为空时间或普通未开始。

### 4.5 搜索

需要提供一个轻量搜索框，支持搜索：

- 队伍名称。
- 队伍缩写。
- 赛事名称。
- 联赛名称。
- 游戏项目展示名。

搜索应与筛选条件组合生效。

### 4.6 排序

至少支持：

- 时间升序。
- 时间降序。

建议支持：

- 状态优先：进行中、未开始、已结束。
- 赛事名称排序。
- 按更新时间。
- 按赛事级别，如果数据源有稳定字段。

默认排序：

- 赛程视图：时间升序。
- 结果视图：时间降序。

### 4.7 数据状态

比赛状态必须在列表和详情中清晰可见：

- 未开始：展示开赛时间。
- 进行中：展示进行中标识和最近更新时间。
- 已结束：展示比分和胜者。
- 延期：展示原计划时间和延期状态。
- 取消：展示取消状态。

如果外部数据源返回未知状态，后端应映射为可展示的兜底状态，前端文案需要避免误导用户。

## 5. 接口需求

### 5.1 查询比赛列表

建议将现有接口扩展为：

```http
GET /api/matches?from=2026-05-01&to=2026-05-24&game=lol&status=finished&query=edg&league=LPL&sort=beginAt_desc
```

兼容要求：

- 继续支持旧参数 `date=YYYY-MM-DD`，等价于 `from=date&to=date`。
- 继续支持旧参数 `game=all|cs2|valorant|lol`。
- 旧调用 `/api/matches?date=YYYY-MM-DD&game=all` 不应被破坏。

新增参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `from` | string | 否 | 北京时间开始日期，格式 `YYYY-MM-DD` |
| `to` | string | 否 | 北京时间结束日期，格式 `YYYY-MM-DD` |
| `status` | string | 否 | `all`、`not_started`、`running`、`finished`、`postponed`、`cancelled` |
| `query` | string | 否 | 队伍、赛事、联赛关键词 |
| `league` | string | 否 | 联赛或赛事名称筛选 |
| `team` | string | 否 | 队伍名称或缩写筛选；可与 `query` 同时存在 |
| `region` | string | 否 | 赛区或国家筛选 |
| `stage` | string | 否 | 赛事阶段筛选 |
| `sort` | string | 否 | `beginAt_asc`、`beginAt_desc`、`status`、`updatedAt_desc`、`league` |

参数语义：

- `query` 是全局关键词，命中队伍、赛事、联赛、游戏展示名即可。
- `team` 是队伍定向筛选，只匹配参赛队伍名称、缩写或别名。
- `league` 在 UI 上可以显示为“赛事”，但接口层保留 `league` 命名以兼容当前数据模型。
- `stage` 如果当前数据源缺失，可先忽略筛选并在响应元信息中返回可用筛选项为空。

### 5.2 响应结构

建议扩展响应元信息：

```ts
type MatchesResponse = {
  date?: string;
  from: string;
  to: string;
  timezone: "Asia/Shanghai";
  filters: MatchFilters;
  sort: MatchSort;
  stale: boolean;
  updatedAt: string;
  total: number;
  facets?: MatchFacets;
  matches: Match[];
};
```

`Match` 建议增加：

```ts
type Match = {
  id: string;
  game: GameType;
  league: string;
  tournament: string;
  serie?: string | null;
  region?: string | null;
  stage?: string | null;
  beginAt: string;
  endAt?: string | null;
  displayDate: string;
  displayTime: string;
  status: MatchStatus;
  bestOf?: number | null;
  teams: Team[];
  result?: MatchResult | null;
  games?: MatchGameResult[];
  streamUrl?: string | null;
  replayUrl?: string | null;
  source: "pandascore";
  updatedAt: string;
};
```

`MatchFacets` 建议返回当前查询范围内可用的筛选项，用于前端下拉：

```ts
type MatchFacets = {
  games: FacetOption[];
  statuses: FacetOption[];
  leagues: FacetOption[];
  teams: FacetOption[];
  regions: FacetOption[];
  stages: FacetOption[];
};
```

详情页或详情弹窗可以复用 `Match`，0.2.0 暂不强制新增独立详情接口。若后续需要更完整的小局、地图、直播、回放数据，可再增加：

```http
GET /api/matches/:id
```

### 5.3 参数校验

必须校验：

- `from`、`to`、`date` 必须是合法 `YYYY-MM-DD`。
- `from` 不得晚于 `to`。
- 日期范围不得超过最大跨度，默认 31 天。
- `game`、`status`、`sort` 必须在白名单内。
- `query` 需要做长度限制，建议 1 到 80 个字符。
- `team`、`league`、`region`、`stage` 需要做长度限制，建议 1 到 80 个字符。
- 文本筛选参数需要 trim，空字符串按未传处理。

错误码建议：

- `INVALID_DATE_RANGE`
- `DATE_RANGE_TOO_LARGE`
- `INVALID_STATUS`
- `INVALID_SORT`
- `INVALID_QUERY`
- `INVALID_FILTER`

## 6. 前端交互需求

### 6.1 页面结构

页面建议调整为：

```text
Header
  站点名称 / 当前数据范围 / 刷新

Toolbar
  视图切换：赛程 / 结果
  日期范围：快捷项 + 自定义日期
  筛选：游戏 / 状态 / 赛事 / 赛区 / 队伍 / 阶段
  搜索：队伍或赛事
  排序
  清空筛选

Main
  比赛列表
  比赛详情展开
  Loading / Error / Empty / Stale
```

如果后续加入“热门赛事”和“我的关注”，应作为同级视图标签出现，不应混入筛选器中。

### 6.2 默认状态

- 首次打开默认进入赛程视图。
- 默认日期范围为今天到未来 7 天。
- 默认游戏为全部。
- 默认状态为未开始和进行中。
- 默认排序为时间升序。
- 默认不启用赛事、队伍、赛区、阶段筛选。

### 6.3 结果视图默认状态

- 日期范围为最近 7 天。
- 状态为已结束。
- 排序为时间降序。

### 6.4 空状态

空状态文案需要区分：

- 当前时间范围没有比赛。
- 当前筛选条件下没有比赛。
- 搜索关键词没有匹配结果。
- 数据源暂未收录该范围比赛。
- 当前日期范围超过允许跨度。

### 6.5 详情交互

- 列表卡片需要提供明确的详情入口。
- MVP 可使用列表内展开详情，避免引入路由复杂度。
- 展开详情后应展示比分、BO、赛事阶段、数据来源和更新时间。
- 没有比分、小局、直播或回放数据时，应显示“暂无数据”或直接隐藏字段，不能显示 `null`、`undefined`。

## 7. 技术实现建议

### 7.1 分阶段实现

建议分三步交付：

1. 日期范围与状态筛选：先打通历史结果查询。
2. 搜索、赛事、队伍筛选：提升定位能力。
3. 赛程/结果视图和详情展开：完善使用体验。

本版本交付顺序应优先保证 P0 能力完整可用。P1 能力如果影响范围过大，应拆到 0.2.x 小版本或 0.3.0。

### 7.2 后端重点

- 扩展参数校验工具。
- 改造 `getMatches` 支持日期范围。
- 缓存 key 从 `matches:{date}:{game}` 扩展为包含范围和筛选条件。
- 先从 PandaScore 拉取指定范围数据，再在服务端做二次筛选。
- 保留旧接口兼容路径。
- 补充比赛状态映射，确保延期、取消、未知状态有稳定前端展示值。
- 对查询范围内的比赛生成 facets，减少前端硬编码筛选项。

### 7.3 前端重点

- 将筛选状态集中管理，避免散落在多个组件中。
- 使用 URL 查询参数表达当前筛选，便于刷新和分享。
- 列表和详情组件继续保持紧凑、可扫读。
- 移动端筛选控件需要避免挤占过多首屏空间。
- 日期快捷项和自定义日期范围要互相同步，避免出现 UI 显示与请求参数不一致。
- 搜索框需要有清空按钮，并且清空后立即恢复当前筛选下的列表。

## 8. 验收标准

0.2.0 完成后应满足：

- 可以查询任意合法单日比赛，不限今天和明天。
- 可以查询最多 31 天的日期范围。
- 可以查看过去比赛结果，并按时间倒序展示。
- 可以按游戏、状态、赛事、队伍关键词组合筛选。
- 可以用全局关键词命中队伍、赛事、联赛和游戏展示名。
- 可以在比赛列表中打开详情，并看到 BO、比分、状态、数据来源和更新时间。
- 延期和取消比赛不会被误展示为普通未开始比赛。
- 旧接口调用方式仍可用。
- 页面刷新后保留当前筛选条件。
- 无结果、加载失败、stale 数据均有明确提示。
- 非法日期范围、非法状态、非法排序会返回明确错误码。
- `npm test` 通过。
- `npm run build` 通过。
- 至少完成桌面和移动端人工验收。

## 9. 暂缓能力

以下能力建议留到后续版本：

- 收藏队伍、赛事或游戏。
- 首页“我的关注”标签。
- 首页“热门赛事”标签。
- 开赛提醒。
- 日历订阅。
- 多数据源融合。
- 直播和回放链接管理。
- 赛事重要程度自定义权重。
- 用户级个性化排序和推荐。
